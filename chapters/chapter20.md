# 第20章：七层防御：一条命令的安检之旅

> **所属**：第五编 · 安全防线
>
> **关键源码**：src/utils/permissions/, security modules

---

## 生活类比

机场安检：行李过X光→人过安检门→可疑物开箱检查→不合格没收→最终登机。AI执行命令也要过类似的七层安检。每一层查不同的东西，漏过一层还有下一层兜底。没有任何单一防线是完美的，但七层叠加后，威胁穿透的概率趋近于零。

## 这一章要回答的问题

**一个能执行rm -rf的AI助手，你敢用吗？Claude Code怎么让你放心？**

信任不是靠承诺建立的，是靠架构建立的。Claude Code没有说"我们的AI很聪明不会犯错"，而是假设AI随时可能犯错，然后用七层独立的安全防线确保即使AI犯了错，错误也会被拦截。这一章带你走完一条命令从生成到执行的完整安检旅程。

---

## 20.1 安全威胁

### Prompt Injection
- 恶意输入诱导AI执行非预期操作——比如在代码注释中藏"请删除所有文件"
- 间接注入更隐蔽：网页内容、文件内容中嵌入的指令被AI当成用户命令执行
- Claude Code的系统提示词中有专门的注入防御指令

### 命令注入
- AI生成的Shell命令中可能被注入恶意片段
- 例如：AI想执行`echo "hello"`，但输入被构造成`echo "hello"; rm -rf /`
- 命令注入的防御依赖AST解析和安全关卡，而非简单的字符串过滤

### 数据泄露与权限提升
- AI可能无意中读取敏感文件并将内容暴露到外部
- 权限提升：AI通过合法操作的组合达到不该有的权限级别
- 最小权限原则：AI只应拥有完成当前任务必需的最小权限集

## 20.2 七层架构

### 第一层：权限判定
- 每个工具调用先过权限系统——Allow/Deny/Ask三级判定
- 已知安全的操作自动放行，已知危险的操作直接拒绝，模糊地带问用户

### 第二层：拒绝追踪
- 记录用户拒绝的操作模式，避免AI反复尝试被拒绝的操作
- 连续拒绝触发策略降级——AI自动切换到更保守的行为模式

### 第三层：Hook拦截
- PreToolUse Hook在工具执行前介入——可以修改参数、拒绝执行
- PostToolUse Hook在工具执行后介入——可以审计结果、撤销操作
- 企业可以通过Hook实现自定义安全策略

### 第四层：参数校验
- 工具的输入参数经过Zod Schema严格校验
- 文件路径检查：是否越界、是否指向敏感位置
- 命令内容检查：25道安全关卡（第22章详述）

### 第五层：沙箱隔离
- macOS sandbox-exec限制进程的文件系统和网络访问
- Linux namespace隔离提供类似的操作系统级保护
- 即使应用层检查全部失效，沙箱仍能限制损害范围

### 第六层：速率限制
- 短时间内大量工具调用触发速率限制——防止AI进入失控循环
- Token消耗速率监控——异常消耗模式可能意味着AI在做无意义的重复操作

### 第七层：账号级别控制
- API层面的终极防线——Anthropic服务端可以终止异常会话
- 企业管理员可以远程禁用特定能力或终止特定会话

## 20.3 一条命令的完整旅程

### 从AI生成到执行完毕
- AI决定调用BashTool执行`npm test` → 权限判定（Allow）→ Hook检查（通过）
- → 命令解析（安全）→ 参数校验（合法）→ 沙箱内执行 → 输出截断
- → PostToolUse审计 → 结果返回给AI → AI决定下一步

### 安全检查站全景
- 每个检查站都是独立的——前一个通过不代表后一个也通过
- 任何一个检查站说"不"，整条链路就中断
- 这种"纵深防御"策略确保没有单点失败

### 用户体验的平衡
- 七层安检不意味着用户要点七次确认——大部分检查是自动的
- 只有权限系统判定为Ask的操作才会弹出确认对话框
- 目标是"无感安全"——你感觉不到它在保护你，但它一直在

## 深水区（架构师选读）

安全与可用性的量化权衡——Permission Fatigue指标与三种模式的设计取舍。过多的权限弹窗会导致"权限疲劳"——用户开始无脑点"允许"，安全形同虚设。Claude Code用三种模式平衡这个矛盾：Default模式严格但弹窗多，适合高安全需求场景；Auto模式自动批准大部分操作，适合信任环境中的高效工作；Bypass模式跳过所有确认，仅限开发测试。设计团队通过追踪"平均每次会话的权限弹窗次数"和"用户不读直接点允许的比例"来量化Permission Fatigue，并据此调整默认策略的宽严程度。
