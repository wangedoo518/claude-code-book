## 11.1 while(true)主循环

### 调用API获取回复
- 将当前消息历史发送给Claude API
- API返回的内容可能是纯文本回复，也可能包含工具调用请求
- 流式接收回复，边接收边渲染到终端

### 解析回复：有工具调用？
- API返回的消息中检查是否包含`tool_use` content block
- 如果有工具调用：提取工具名称和参数，准备执行
- 如果没有工具调用：AI的回答就是最终结果

### 执行工具 → continue / 无工具 → break
- 有工具调用：执行工具→将结果追加到消息历史→continue（回到循环顶部）
- 无工具调用：break退出循环→将最终回复展示给用户
- 这就是整个Agent Loop的核心逻辑——惊人的简洁

## 11.2 5种停止条件

### 无工具调用（正常完成）
- AI认为任务已完成，回复纯文本——这是最常见的退出路径
- AI主动选择"不再调用工具"就意味着"我做完了"

### maxTurns限制
- 预设的最大循环次数，防止AI陷入无意义的工具调用循环
- 超过限制后强制退出，向用户报告"已达到最大轮次"

### 用户中断（Ctrl+C）
- 用户随时可以按Ctrl+C中断当前循环
- 中断是安全的——已完成的操作不会被回滚，但正在执行的工具会被取消

### 错误退出
- API调用失败、工具执行异常、不可恢复的错误
- 错误信息会被记录并展示给用户，Agent Loop安全退出

### Token耗尽
- 上下文窗口已满，无法再追加新消息
- 触发上下文压缩或提示用户开始新会话

## 11.3 TaskBudget控制

### 为什么限制循环次数
- AI可能陷入"读文件→发现问题→修改→发现新问题→再修改"的无限循环
- 没有预算限制，一个简单任务可能消耗数百次API调用

### 无限循环检测
- 检测连续多次调用相同工具、相同参数的模式
- 检测AI反复在相同文件上做相似修改的行为
- 这些模式通常意味着AI"卡住了"，需要人工介入

### 安全阀与真实多轮次示例
- TaskBudget设置了硬性上限和软性警告
- 一个真实场景：用户要求"重构这个模块"——AI读取5个文件、修改3个、运行测试、修复2个失败测试——共12个Turn，全部在预算内自动完成
- 安全阀的意义：让AI自主工作时不会失控

## 深水区（架构师选读）

Agent Loop的学术基础是ReAct（Reasoning + Acting）范式——AI交替进行推理和行动。在实现层面，query.ts处理了一个重要的技术细节：`tool_use` content block的流式组装。API以SSE事件流返回回复，工具调用的参数可能跨越多个`content_block_delta`事件。StreamingToolExecutor需要将这些碎片正确拼接成完整的JSON参数后才能开始执行。此外，当AI在一次回复中请求多个工具调用时，如果这些工具之间没有数据依赖，Claude Code会尝试并行执行以提高效率。
